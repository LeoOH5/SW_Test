## 🟠 중요 (Major)

### M-1. 초장문 파일명 업로드 시 서버 500 (미처리 예외)
- **절차**: 파일명이 한글 120자(`가`×120)`.jpg`인 정상 이미지를 360이미지로 업로드
- **기대**: 파일명 길이 초과 시 `400`(잘못된 요청)으로 안내
- **실제**: `POST upload.visitinc.kr/scene/autoNorth` → **`500 Internal Server Error`** + 콘솔 500. UI에는 "업로드 요청에 실패했습니다"만 표시
- **근본원인 가설**: 한글 200자≈UTF-8 약 600바이트 → 파일시스템 NAME_MAX(255) 또는 DB 컬럼 길이 제약 초과를 서버가 검증 없이 처리하다 예외. (정상 서버라면 400)
- **증거**: `recon/upload_test.json` file15 · 재현율 관찰상 100%

### M-2. 로그인 실패 응답에 사용자 열거(user enumeration) + 부정확한 상태코드
- **절차**: 존재하지 않는 이메일(정상형식) + 임의 비밀번호로 `POST vtapi.visitinc.kr/user/login`
- **기대**: 실패는 `401`, 메시지는 "아이디 또는 비밀번호가 올바르지 않습니다"처럼 **계정 존재 여부를 숨김**
- **실제**: **`HTTP 201` + `{"message":"id not exist"}`** → ① 로그인 실패에 201 반환(부정확) ② "없는 계정"임을 그대로 노출 → 공격자가 어떤 이메일이 가입돼 있는지 열거 가능
- **증거**: `recon/additional_tests2.json` A_없는계정_정상형식

### M-3. 보안 응답 헤더 전면 부재
- **절차**: `https://roundpic.io/` 응답 헤더 확인
- **기대**: CSP, X-Frame-Options(또는 CSP frame-ancestors), HSTS, X-Content-Type-Options 등 기본 보안 헤더 존재
- **실제**: CSP · X-Frame-Options · Strict-Transport-Security · X-Content-Type-Options · Referrer-Policy · Permissions-Policy **6종 모두 없음**
- **영향(정직하게)**: 클릭재킹·MIME 스니핑 **가능성**(실증 iframe PoC는 사람 확인 필요). HSTS 부재는 관찰된 단일 200 응답 기준(엣지/리다이렉트 레이어 별도 확인 권장)
- **증거**: `recon/recon.json` 보안헤더 present:[]

## 🟡 경미 (Minor)

### m-1. 업로드 동기 게이트가 확장자만 신뢰 (콘텐츠 검증 여부 미확정)
- 텍스트 파일을 `.jpg`로 위장(`12_fake_text_as.jpg`) → 동기 게이트가 **202 수락**. 반면 `.webp`/`.bmp`는 동기적으로 code203 거부.
- 즉 **동기 게이트는 확장자 기반**으로 보임. 다만 202는 비동기 처리 시작을 뜻하므로 **매직바이트/콘텐츠 최종 거부 여부는 사람 확인 필요**(단정하지 않음).
- 증거: `recon/upload_test.json` file07,09,12

### m-2. 허용 포맷 목록 불일치 (사양 확인 요망)
- `gif`는 202 수락, `webp`·`bmp`는 code203 거부. (PNG는 무손실 파노라마로 정상이므로 결함 아님)
- 파노라마 저작도구에서 애니메이션 gif 허용이 의도인지 **사양 확인 필요**. 결함으로 단정하지 않고 관찰로 보고.
- 증거: `recon/upload_test.json` file06~09

### m-3. 파노라마 비율 검증 없음 (게이트 확정 / 렌더 결과 미확정)
- 정사각(1:1)·16:9·세로 이미지가 동기 게이트 통과(202). equirectangular는 2:1이어야 함.
- **뷰어에서 실제로 왜곡되는지는 사람 시각 확인 필요**.
- 증거: `recon/upload_test.json` file02~04

### m-4. 0바이트·손상 파일 게이트 통과 (최종 처리 미확정)
- `10_zero_bytes.jpg`(0B), `11_corrupt_truncated.jpg`(잘린 jpg)가 동기 게이트 202 통과.
- 비동기 처리에서 최종 거부되는지는 사람 확인 필요.
- 증거: `recon/upload_test.json` file10,11

### m-5. 없는 라우트 처리 부재
- 존재하지 않는 해시 라우트 접근 시 정상 404/리다이렉트 대신 **처리되지 않은 콘솔 에러(Angular NG04002)**.
- 증거: `recon/recon.json` 콘솔에러

### m-6. 홍보링크(website) URL 검증 없음
- 스페이스의 홍보링크 필드에 URL이 아닌 값(`not_a_valid_url_링크`)이 저장됨 → 형식 검증 부재.
- 증거: `recon/verify_state.json` cnt153 website

### m-7. welcomeMessage에 문자열 "undefined" 저장 (추정)
- 두 스페이스 공통으로 welcomeMessage 값이 문자열 `"undefined"`. 프론트에서 JS `undefined`가 문자열로 강제 변환돼 저장된 것으로 **추정**(입력 경로 미확인).
- 로그인 검증 오류 메시지가 비밀번호 정책(6~20자)을 상세 노출하는 점도 함께 관찰(정보 노출 경미).
- 증거: `recon/verify_state.json`, `recon/additional_tests2.json`

## ✅ 정상 (PASS) — 커버리지 증빙

- **권한/IDOR 차단**: `company?cmpnId=` 를 미소속 값(1,2,50,99,150,200,300)으로 바꿔도 `code103 "not a member"`로 차단. 접근된 건 내 소속 회사(100·101, 둘 다 userId:50 소유)뿐. 인접 ID 99가 차단되어 **순차 IDOR도 반증**됨.
- **비인증 접근 차단**: 토큰 없이 API 호출 시 `401 "token is not valid"`.
- **서버측 입력검증 존재**: 로그인에 빈값/공백/형식오류/비번 길이 위반 → `400`(class-validator).
- **테스트 스페이스 비노출**: 두 스페이스 모두 `active=false` (실서비스 메인 미노출) 확인.
