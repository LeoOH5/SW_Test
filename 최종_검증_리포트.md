> 대상: roundpic.io(웹) · vtapi.visitinc.kr(API) · upload.visitinc.kr(업로드) · business.roundpicapp.com(스토리지)
> 작성: 2026 중부권 SW테스트 경진대회 참가팀 (오정빈 — 수동/UI · 정해찬 — 자동/API)
> 일자: 2026-07-19
> 범위: 개발사 제공 테스트 계정(회사 cmpnId=100·101)으로 수행한 인가된 테스트. 안전수칙 준수 — 스페이스 active=false 유지, 생성 데이터 즉시 정리, 변경 필드 원복, 계정잠금·타 고객·DoS 미수행.
> 목적: 발견 결함이 현재도 실제로 재현되는지 자체 검증하고, 재현·증적이 확실한 항목만 심각도와 함께 확정한다.

## 1. 결론 요약 (Executive Summary)

- 확정 결함: 심각 2 · 보통 7 · 경미 26 · 개선 2 = **표기 항목 37건**. 이 중 4건(N-3·N-5·E-011·E-014)은 상위 항목의 확장·동반 항목이므로, **고유 결함으로 집계하면 33건**이다(2장 각주 참조).
- 최우선(개발사 즉시 검토 권고)
  1. 미게시 스페이스 비인증 정보 유출 (심각, A-01) — 게시 전 콘텐츠 메타데이터·썸네일이 로그인 없이 노출.
  2. 배치 업로드 데이터 손실 (심각, E-018) — 미지원 포맷 1장 혼입 시 같은 배치의 정상 파노라마까지 전량 유실.
- 검증 방식: 자동(API 직접 관찰)과 수동(브라우저 육안)을 교차해 재현 여부·근본원인·정확한 심각도를 함께 확정. 이번 검증에서 저장형 XSS 후보(N-1)를 육안으로 하향하고, 미게시 썸네일 노출(N-3)을 육안으로 확정했다.
- 정직성: 테스트했으나 정상(PASS)인 항목(9장)과 안전범위상 미수행한 고위험 항목(10장)을 모두 명시해 오탐·과대평가를 배제한다.

## 4. 심각(Critical) 결함 상세

### A-01. 미게시(active=false) 스페이스 콘텐츠가 비인증 API로 유출
- 재현: active=false 스페이스에 씬을 생성한 뒤 토큰 없이 `GET https://vtapi.visitinc.kr/vt?VT=<7자리 hex>` 호출 → HTTP 200으로 스페이스 제목·씬 정보·360 타일 저장 경로·원본 업로드 파일명을 반환. (자동 실측 당시 응답 예: `contentTitle:"SWTEST_QA_스페이스"`, `scene.title:"00_normal.jpg"`.)
- 육안 재확인(2026-07-19): 씬을 정리(삭제)한 현재 상태에서 시크릿탭으로 동일 엔드포인트 호출 시 `{"status":false,"message":"not exist defaultScene","code":103}` — 로그인 없이 서버가 응답을 처리함을 확인(인증 게이팅 부재). 콘텐츠 유출 증거는 씬이 존재하던 자동 실측 캡처가 정본이다.
- 정밀화(정직성): 360 타일 본체 URL(`business.roundpicapp.com/storage/VT/...`, 서명 URL)은 비인증 403으로 보호됨 → 타일 이미지 대량 다운로드는 불가. 그러나 미게시 콘텐츠의 존재·구성·경로·썸네일·원본 파일명 메타데이터가 공개된다. (썸네일 노출은 N-3에서 육안 확정.)
- 영향: 게시 여부와 무관하게 VT 키만 알면 미공개 스페이스의 메타데이터·미리보기를 조회. 키가 7자리 hex라 열거 가능성 존재(열거 범위는 안전·법적 이유로 미검증, 10장).
- 수정 제안: `/vt` 응답을 게시상태(active)로 게이팅, 썸네일도 동일 게이팅, VT 키 엔트로피 강화.
- 증적: `recon/verify_all_fresh.md`, `recon/full_audit.json`(A01X), `recon/A01_manual_api.png`.

### E-018. 배치 업로드에 미지원 포맷 1장 혼입 시 정상 파일까지 전량 유실 (데이터 손실)
- 재현: 다중선택 배치에 거부 포맷(WebP) 1장만 섞여도 같은 배치의 정상 파노라마까지 전부 등록 실패. 실패 사유 안내 없이 "업로드 실패" 토스트 + 빈 유령 카테고리 잔존.
- 통제 실험(8종 매트릭스, 콘텐츠·용량·개수·순서 교란 전부 제거): 거부 파일(빈 jobId)이 1장이라도 포함된 배치는 정상분까지 전량 실패. 반면 해상도실패만 있는 배치(#2)는 정상분 생존(부분 성공) — 즉 트리거는 "거부 파일 포함"에 한정되며, 이 조건에서의 재현은 일관됨. 파일 개수·용량·동시성·업로드 순서는 모두 반증됨.
- 근본원인(자동 API 교차검증으로 로커스 시사): 직접 API 호출에서 정상 씬(scnId 1894)이 거부 업로드 후에도 서버에 유지됨을 확인(`verify_v5.json` E-018_4, 판정 "API레벨_정상씬_유지"). 이는 관측된 사실이며, 따라서 정상분 유실은 서버 롤백이 아님이 확인된다. 원인은 클라이언트(프론트엔드 배치 완료 핸들러)에 있음이 강하게 시사되고, 구체적 메커니즘은 거부 파일의 빈 jobId(code203)를 집계하다 성공분 씬 확정에 실패하는 것으로 추정한다(정확한 지점은 개발사 소스 확인 필요).
- 트리거 조건: WebP는 게이트에서 빈 jobId 반환이 트리거. (GIF는 게이트 통과 후 비동기 거부라 트리거 여부 별도 확인 권장.)

- 수정 제안: 배치 완료 핸들러가 개별 파일 실패를 격리 처리(부분 성공 허용) + 실패 사유·대상 파일 고지. 클라이언트 업로드 큐에서 빈 jobId 방어.
- 증적: 통제 매트릭스(`VT_자동테스트_리포트_v5_병합.md` §E-018), `recon/verify_v5.json`(E-018_1~4).

## 5. 보통(Major) 결함 상세

- **A-02. 로그인 사용자 열거 + 실패에 성공 코드**: 없는 계정 → `201 {"code":101,"message":"id not exist"}` / 가입계정+틀린비번 → `201 {"code":102,"message":"pwd not compare"}`. 응답 코드로 계정 존재 여부가 구분되어 사용자 열거가 가능하며, 인증 실패를 2xx로 통보. 수정: 실패 시 401 + 단일 일반 메시지. 증적: `recon/verify_all_fresh.md`(A-02 행).
- **A-03. 입력검증 미비로 4xx 대신 500**: 초장문 파일명(한글 100자 이상) 업로드 → `POST /scene/autoNorth` 500. `PUT /content`에 존재하지 않는/불완전 body → 500(생성 `POST /content`는 400 검증 존재 → 검증 비대칭). 500 본문은 스택 미노출(양호). 수정: 전 엔드포인트 일관된 입력검증(400)·예외 핸들링. 증적: `recon/verify_all_fresh.md`(A-03 행), `recon/full_audit.json`(INFO-1).
- **A-04. 보안 헤더 6종 전무 + 토큰 평문 저장**: 웹·API 모두 CSP·X-Frame-Options·HSTS·X-Content-Type-Options·Referrer-Policy·Permissions-Policy 부재, `x-powered-by: Express` 노출. accessToken·refreshToken이 localStorage 평문 → 헤더 부재와 결합 시 XSS 발생 시 세션 탈취 표면 확대. 수정: 표준 보안 헤더 세트 적용, `x-powered-by` 제거, 토큰은 httpOnly+Secure 쿠키 검토. 증적: `recon/verify_all_fresh.md`(보안헤더 행), `recon/full_audit.json`(AUTH-5).
- **E-019. 특정 해상도 파노라마 처리 실패(콘텐츠 무관)**: 정상 규격(2:1·유효 JPEG)인데 약 10771×5385는 실패(jobId 발급 후 씬 미등록), 7180×3590·11968×5984는 통과. 해상도만 스왑하면 성패 반전(A/B 통제 + 자동 재현 확정). 수정: 처리 가능 해상도 상한 명시·사전검증·실패 사유 고지. 증적: `recon/verify_v5.json`.
- **E-009. 초소형(100×50) 이미지가 씬으로 통과 → 뷰어 검은 화면**: 최소 크기/비율 검증 부재. API에서 tiny 씬 생성 + 뷰어 완전 검은 화면(양쪽 확인). 안내·플레이스홀더 없음. 수정: 최소 해상도·비율 검증 추가. 증적: `recon/V2_EVIDENCE.md` §D + 육안.
- **E-010. 업로드 용량 상한이 안내와 5배 이상 불일치**: 안내 "5MB까지"인데 실측 24MB까지 통과, 35MB 거부. 상한 미정의·안내 오류. 수정: 실제 상한 정의·안내 일치. 증적: 수동/육안(오정빈).
- **E-012. 업로드 직후 에디터가 "이미지 없음"으로 오표시(새로고침 필요)**: 비동기 job 반영 지연. 육안 + 자동(렌더 캡처 차단) 양쪽 확인. 수정: 업로드 완료 이벤트로 뷰 갱신. 증적: 수동/육안 + 자동 부분.

1. 계정 잠금 / 로그인 rate limit 부재 여부(A-02 열거와 결합 시 브루트포스) — 테스트 계정 잠금 위험.
2. 객체단위 IDOR(scnId/hsId/cntId/rsviId/ucmpnrId 타인 객체 조회·수정·삭제) — 제2계정 필요, 쓰기 IDOR는 데이터 파괴 위험.
3. VT 키 열거 범위(7자리 hex로 타 고객 미게시 스페이스 접근 가능성) — 실 타 고객 데이터라 법적·윤리 위험, 개발사 입회 필요.
4. Mass assignment(`PUT /content`/`/company`에 active:true·cmpnId·role 주입으로 게시상태·소유권·권한 강제) — active=false 유지 위반 위험으로 미수행. 성공 시 A-01 게이팅·회사제한·격리 우회 → 최우선 협의 대상.
5. 디코딩 DoS(픽셀폭탄 초고해상도) · 회사 10개 제한 우회 · active 토글 인가 · 경쟁상태 중복생성.
6. 저장형 XSS 뷰어 실행(N-1) — `PUT /content`가 유효 cntId에도 500이라 API로 뷰어 필드 주입 불가 → 에디터 UI로 사람이 실행 검증 필요.

## 11. 방법·정직성
- 자동: Playwright(헤드리스+CDP), 실 API 직접 관찰. 수동: Chrome 실사용·육안·콘솔·통제 실험. 완전성 크리틱으로 미커버 공격면 30종 도출 후 안전 18종 실측, 실행 여부가 갈리는 항목은 육안 최종 판정.
- 정직성 원칙: 재현·증적이 확실한 항목을 중심으로 등재하되, 미수행·부분 항목은 검증 상태에 명시했다. 미재현(배치 white screen)은 경미로 하향, 기수정(공유버튼·폼잔존, E-006/007) 제외, 추정은 "추정"·"시사"로 명기, 세션 반복만료는 동시 로그인 원인으로 결함 제외, N-1은 육안 확인 후 하향, A-05/A-06/A-07은 이번 실행에서 미재확인임을 검증 상태에 명시(과거 V2 증거 근거).
- 안전 마무리: 생성 씬 전부 삭제(남은씬=null), 회사명·이메일 변경분 원복(cmpnId 100 SW테스트_회사01 / cmpnId 101 SW테스트_회사02, 이메일 swtest01@example.com), 최종 3개 스페이스 active=false 확인.
