# VT partners — 테스트 설계 산출물 (기술성 20점 근거)

> 공식 기법 10점 + 창의적 기법 10점을 "설계 산출물"로 증명하기 위한 문서.
> 결과보고서(붙임2)의 각 TC "소분류"와 아래 설계표를 연결해 제시.

---

## 1. 기능 트리 / 인벤토리 (커버리지 근거)

```
1. 계정/인증   : 로그인 · 이메일/PW 검증 · 비밀번호 표시토글 · 비밀번호찾기 · 문의
2. 회사/스페이스: 회사 목록 · 스페이스 생성/수정/삭제 · 발행 토글 · 홍보링크·메모
3. 파노라마/씬 : 이미지 업로드 · 씬 생성/삭제 · 순서/이름 · 카테고리 탭 · 지도 좌표
4. 핫스팟      : 씬이동 · 사진 · 동영상 · 예약
5. 뷰어        : 360 회전/줌 · 씬 이동 · 씬 목록/탭 · 지도 · 공유 · 오류처리(잘못된 ID)
6. 비기능      : 브라우저 호환 · 콘솔/안정성 · 접근성(lang) · 성능(스크립트 로딩)
```

실제 서비스 구조: **회사(Company) → 스페이스(Space/VT) → 씬(Scene) → 핫스팟**. 뷰어는 로그인 불필요, 관리자는 계정 필요.

---

## 2. 공식 테스트 기법 (10점) — 설계표

### 2-1. 동등 분할 (Equivalence Partitioning) — 로그인 이메일
| 클래스 | 예시 입력 | 기대 | 실제 |
|---|---|---|---|
| 유효(정상) | user@visitinc.kr | 통과 | 통과 |
| 무효-형식 | notanemail (@없음) | 형식오류 차단 | **통과됨(결함 E-004)** |
| 무효-빈값 | (빈 문자열) | 필수오류 | **오류 미표시(E-004)** |

### 2-2. 경계값 분석 (Boundary Value) — 비밀번호 길이 & 파노라마 해상도
비밀번호 규칙 "6자 이상 21자 미만"(=6~20):
| 지점 | 값 | 기대 |
|---|---|---|
| min-1 | 5자 | 무효 |
| **min** | **6자** | **유효** → 실제 유효(ng-valid)이나 오류문구 잔존(E-005) |
| max | 20자 | 유효 |
| max+1 | 21자 | 무효 |

파노라마 이미지 해상도(회원 준비 세트로 실측):
| 지점 | 값 | 기대 | 실제 |
|---|---|---|---|
| 극소(하한 밖) | **100×50** | 거부 또는 정상 | **검은 화면(E-001, 심각)** |
| 정상 | 규격 파노라마(예 4096×2048) | 정상 | 정상 |
| 대용량(상한 근처) | 5MB 초과 | 처리/거부 명확 | 업로드 허용(렌더 상세 재확인 필요) |

### 2-3. 결정 테이블 (Decision Table) — 뷰어 접근 권한/오류처리
| 조건: VT ID 유효 | 조건: 발행 상태 | 조건: 로그인 | 기대 액션 | 실제 |
|---|---|---|---|---|
| 유효 | 발행 | 무관 | 뷰어 정상 표시 | 정상 |
| 유효 | 미발행 | 비로그인 | 접근 제한/안내 | (추가확인) |
| **무효** | - | 비로그인 | **오류/404 안내** | **관리자 저작화면 노출(E-002)** |

### 2-4. 상태 전이 (State Transition) — 스페이스 생명주기
```
[없음] --생성--> [작성중/미발행] --이미지업로드--> [콘텐츠보유]
   [작성중] --발행토글 ON--> [발행] --수정--> [발행/작성중] --삭제--> [없음]
```
- 검증 포인트: 미발행 스페이스의 뷰어 접근, 발행 후 공개, 삭제 후 링크/목록 잔존 여부.
- 파생 결함: 잘못된 상태(존재하지 않는 ID)에서 [작성중] 빈 화면으로 폴백(E-002).

### 2-5. 유스케이스 기반 — "VT 하나를 만들어 배포"
회사 선택 → 스페이스 등록 → 에딧모드 → 파노라마 업로드 → 씬/핫스팟 구성 → 발행 → 뷰어 공유. (각 단계에서 위 기법 적용)

## 3. 창의적 테스트 기법 (10점) — 2개 심화

### 3-1. 경계·이상 이미지 세트 기반 "결함 유발형" 업로드 테스트
정상 파노라마 1장을 기준으로 **의도적으로 규격을 벗어난 변형본 세트**를 만들어 업로드 방어를 체계적으로 공략. (회원 준비 스페이스 `업로드_테스트(변형이미지)`에 실제 세트 존재)

| 변형본 | 노리는 결함 | 결과 |
|---|---|---|
| 06_초소형_100x50 | 최소 해상도 방어 | **검은 화면(E-001)** |
| 07_포맷.png | 포맷 필터링 | 정상(허용·렌더) |
| 12_경계_5MB초과 | 용량 상한 | 업로드 허용(추가확인) |

| (제작 권장) 0바이트/손상/확장자위장/비2:1 | 오류처리·검증 | 본선 대비 확장 |

> 논리: "정상 케이스로 기준선(PASS)을 잡고, 경계·이상값으로 방어 부재를 드러낸다."

### 3-2. 개발자도구 기반 상태 관찰(콘솔·네트워크·문서속성) 주입 테스트
UI에 드러나지 않는 결함을 개발자도구로 계측:
- **콘솔 로그 감시** → 반복 런타임 에러 `getVisibleChildId`(E-008) 포착
- **문서 속성 검증** → `document.title`이 파일명으로 고정(E-003), `<html lang="en">`(E-009)
- **잘못된 라우트 콜드 로드** → SPA 오류처리 부재(E-002)
- **네트워크/성능 경고** → Maps API 비동기 미적용(E-010)

> 논리: "눈에 보이는 화면만이 아니라, 콘솔·DOM 속성·라우팅까지 계측해 숨은 결함을 근거와 함께 잡는다."

### (대안) 페어와이즈 조합 — 본선 확장용
브라우저(Chrome/Edge/Firefox) × 해상도(데스크톱/태블릿/모바일) × 핫스팟(씬이동/사진/동영상/예약) 전조합 대신 페어와이즈로 축소 설계. (예선은 Chrome 단일로 진행, 본선 다브라우저 시 적용)
