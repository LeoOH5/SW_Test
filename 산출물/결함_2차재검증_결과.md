## 실측 증거 (2026-07-19)

### Major

**M-1 — 초장문 파일명 500**
```
POST upload.visitinc.kr/scene/autoNorth (cntId=169, 정상 JPEG 64x32)
  filename="short.jpg"          → 202 {accepted:true}
  filename="가"×120+".jpg"      → 500 {"statusCode":500,"message":"Internal server error"}
```
파일 내용은 동일, 파일명 길이만 유일 변수 → 서버 미처리 예외. 정상이면 400.

**M-2 — 로그인 사용자 열거 + 잘못된 상태코드**
```
없는 계정(정상형식) → HTTP 201 {"status":false,"message":"id not exist","code":101}
존재 계정+틀린 비번  → HTTP 201 {"status":false,"message":"pwd not compare","code":102}
```
실패인데 201(401이어야 함). 101 vs 102 메시지·코드가 달라 이메일 가입 여부 열거 오라클 성립.

**M-3 — 보안 헤더 부재 + 평문 HTTP**
```
GET https://roundpic.io/ → 200, 헤더: Accept-Ranges/ETag/Last-Modified/Content-Type/Content-Length/Date 뿐
  CSP · X-Frame-Options · HSTS · X-Content-Type-Options · Referrer-Policy · Permissions-Policy → 6종 전무
GET http://roundpic.io/  → 200 (HTTPS 리다이렉트 없음, 평문 그대로 서빙)
```

### Minor

**m-1 — 콘텐츠 미검증(메커니즘 정정 재확인)**
```
텍스트 내용 + filename .jpg + part-mime image/jpeg      → 202 accepted:true  (위장 통과)
진짜 JPEG 내용 + filename .txt + part-mime text/plain   → 202 code203 "not supported"
진짜 JPEG 내용 + filename .jpg + part-mime octet-stream  → 202 code203 "not supported"
```
판단 기준은 **파일 내용도 확장자도 아닌 클라이언트가 선언한 multipart MIME**. 공격자가 MIME을 image/jpeg로 임의 지정하면 임의 바이트 통과. → v4의 "확장자 기반" 기술은 부정확, v5의 "MIME만 신뢰" 정정이 정확함.
