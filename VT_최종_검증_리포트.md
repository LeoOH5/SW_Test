## 1. v5 신규·재분류 결함 자동 재검증 (2026-07-19 신선 실측)

| ID | v5 주장 | 신선 증거 | 판정 |
|---|---|---|---|
| **E-018** 배치 데이터손실(심각) | 배치에 거부파일(webp) 1장 → 정상 파일까지 전량 유실 | webp=`code203`(빈 jobId) 트리거 재현. 정상 업로드→씬 1894 등록→webp 거부 후 **정상 씬 API 레벨 생존** | ⚠️ **트리거 재현+ / 롤백은 클라이언트측** |
| **E-019** 해상도 처리실패(보통) | ≈10771×5385 실패, 7180×3590 통과 (해상도만 원인) | 10771×5385 → jobId 발급되나 **씬 미등록**, 7180×3590 → **씬 등록(1895)**. 동일 2:1·유효 JPEG | ✅ **자동 완전 재현** (A/B 정확 일치) |
| **E-020** 포맷 정책 불일치(경미) | accept=image/* 허용하나 서버는 JPEG/PNG만 | webp → `code203` 거부(jobId 없음), **gif → 게이트 통과(jobId·accepted)** | ✅ webp 재현 / gif는 정밀화 |
| **E-022** 회사 이메일 검증부재(경미) | `@`없는 문자열이 검증 없이 저장 | PUT `email:"invalidemail_noat"` → **200 status:true**, 저장 확인, 원복 완료 | ✅ **자동 완전 재현** |

### E-018 근본원인 특정 (검증의 핵심 성과)
직접 API 호출에서는 **정상 씬이 거부파일 업로드 후에도 서버에 그대로 남는다**(scnId 1894 생존). 즉 정빈이 UI에서 관측한 "정상 파일까지 전량 유실"은 **서버 롤백이 아니라 프론트엔드 배치 완료 핸들러의 버그**(거부의 빈 jobId를 집계하다 깨지며 성공분 씬을 미확정/폐기)로 로커스가 특정된다.
→ **결함은 실재**(정빈 통제실험 8종이 정본 증거), **원인은 클라이언트측**. 수정 지점을 개발사에 더 정확히 지목 가능.

### E-020 정밀화
정빈은 "webp/gif = 서버가 빈 jobId로 거부"라 했으나, 실측상 **gif는 게이트를 통과(jobId 발급·accepted)** 후 비동기 단계에서 폐기된다(A-06/A-07 계열). **빈 jobId 거부는 webp에 한정**. E-018 트리거도 webp가 정본. (gif 혼입 시 동작은 별도 확인 권장 — 게이트 통과분이라 배치 롤백 유발 여부 다를 수 있음.)

## 5. 안전·정직성
- 생성 씬 전부 삭제(각 단계 남은씬=null), 회사 이메일 변경분 **원복 완료**(swtest01@example.com), 최종 3개 스페이스 active=false 확인.
- 이번 실측에서 **직접 관측한 것만** "재현"으로 표기. 미수행(A-06/07/12)은 "결함 없음"이 아니라 재확인 안 함.
- E-018은 "정상분 유실" 자체를 API로 재현 못 했음을 숨기지 않고 명시 — 대신 **원인을 클라이언트측으로 특정**해 정빈 결함을 뒷받침·강화.

## 6. 재검증으로 강화된 제출 포인트
1. **E-019·E-022** — 자동 독립 재현으로 신뢰도 확정. xlsx 그대로 제출 유효.
2. **E-018** — 근본원인=프론트엔드 배치 핸들러(서버는 정상 씬 유지)로 지목. 개발사 보고 시 수정 지점 명확화.
3. **E-020** — gif는 게이트 통과(webp만 빈 jobId)로 정밀화 권장.
4. **A-01**(심각) — 재현+ 확정, 원본 파일명 노출(E-014)까지 같은 응답에서 확인. 개발사 우선 보고.
