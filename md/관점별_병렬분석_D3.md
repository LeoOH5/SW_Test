## 0. 관통 결론 (3관점 공통)
1. **E-006·E-007은 이미 정상화 → 결함서 제외.** 이 둘의 공통 패턴("클릭/조작 후 상태 UI가 기대와 다르게 보임")을 가진 **E-013·E-012·E-011·E-016을 최우선 재검증**해야 한다(같은 위험군, 이미 수정됐을 수 있음). → E-013은 D3 실측에서 **여전히 재현됨** 확인, E-012/E-011/E-016은 업로드·씬조작 필요로 미완.
2. **세션 만료 = 동시 로그인 충돌 → 결함 아님**(단독세션 저장 정상 확인).
3. 제출 전 **전 결함 재검증 필수** + 심각도 과장 금지 + 결함마다 "명세 기대 vs 실제" 1문장 고정(개발사 판정 방어).

## 1. QA 관점 — 커버리지·재검증·TC 품질

### 1-A. 커버리지 완전 0건 → 우선 채울 소분류 (PASS 19→30+)
**최우선 4종**: 비밀번호 찾기 · 회사 수정 · 스페이스 삭제(✅D3 완료) · 핫스팟 4종(공지/이미지/비디오/예약) 개별 생성.
기타: 로그아웃 정상흐름, 솔루션 이용문의, 전화번호 형식검증, 회사 삭제, active 토글(경고문구까지만·R2), 뷰어 프리뷰 아이콘, 지도(좌표), 브라우저 호환(Edge 1회), 반응형(창축소).

### 1-B. 재검증 우선순위
- **High(강등·재현실패 위험)**: E-013(✅재현확인)·E-012·E-011·E-008(재현조건 미확정=심각근거 취약, 실측 최우선).
- **Med**: E-016(경미↓/제외 검토)·E-005(H스페이스 미확인)·E-009·E-010(증적 재확보)·E-002(✅삭제토스트로 강화)·E-004.
- **Low(D3 재확인 완료)**: E-001·E-003·E-014(공개뷰어만 남음)·E-015 / 제외: E-006·E-007·세션.

### 1-C. TC 완성도 — 현재 가장 약한 2가지
- **기법 태그 열**(각 TC에 경계값/동등분할/결정표/상태전이/API주입/퍼징 태그 → 기술성 산출물과 1:1)
- **PASS 전량 기록**(FAIL만 적으면 미커버로 오인). 그 외: 분류 문구 일치, 전제조건 명시, 기대결과 구체화, 결함ID 1:1, 관련이미지 매핑, 요약·환경 섹션, R6 양식.

### 1-D. 추가 발굴 포인트
토스트 문구 전수 비교(E-002 확장·✅부분확인), 도움말 전반 구식화(E-003 확장), 씬삭제 후 참조 무결성, 업로드 안내문 존재여부(“5MB” 출처 캡처), 예약 과거날짜 허용, 핫스팟 “수정불가” 사전고지 여부, 수정 후 목록 즉시반영.

## 2. TE 관점 — 심각·보통 결함 확정 로드맵 (50점 승부처)

### 2-A. 결함별 재현·증적·방어논리 (업로드 담당자 관찰 체크리스트)
- **E-008 크래시(심각후보·최우선)**: 빈 스페이스에 1장씩 순차 업로드+F12 상시. 가설 3종(업로드 순번/소형↔대형 연속/포맷 혼합) 각 3회. 증적=콘솔 전체스택+white screen+재현율(n/m). 방어=다른 무효파일은 "실패 토스트"로 정상 안내되는데 **이 케이스만 무안내 전체정지=진행불가(심각 정의 부합)**. 간헐성은 "예측불가 크래시"로 재프레이밍.
- **E-009 비율검증부재(보통)**: 정사각/16:9/세로 개별 업로드→뷰어 왜곡·검은화면. 100×50 검은화면과 결합.
- **E-010 용량상한(보통·확정)**: 이진탐색으로 정확 임계값. 업로드 전 **파일 실제 바이트 확인**(파일명≠크기).
- **E-012 업로드직후 빈상태(보통)**: 토스트 성공 vs 화면 미반영=상태동기화 결함(중복업로드 유발).
- **E-016 대표삭제 렌더지연(경미~보통)**: E-012와 동일계열(로딩스피너 부재)로 묶어 서술.

### 2-B. 경계값·페어와이즈 설계
- **용량 BVA**: 06(pass)→12→18→29.5→(분기)32/27→<1MB까지. 확정 24pass/35reject 사이 임계 특정 + 06~24 단조성 검사.
- **비율×포맷×용량 페어와이즈**: 60→13케이스(정상/경계24MB/초과30MB × 2:1/1:1/16:9/세로 × jpg/png/webp/gif/bmp), 각 단독 업로드.

### 2-C. 안전한 보안 시나리오 (R3·R4, `<b>`한도)
로그아웃 후 뒤로가기 관리화면 접근, 세션만료 후 저장 처리, 다중탭 세션전파, **인접 cntId 읽기전용 GET 열거**(파괴X, 소유권 검증 확인), cntId 이상값(음수/문자열) 주입→500/스택노출, Stored XSS `<b>`(회사명·홍보링크·메모·씬타이틀), 비번찾기 사용자 열거, 파일명 XSS.

### 2-D. 성능(P4, 37씬)
지표: 초기로딩·씬전환(10회 평균/최대)·프리즈·콘솔에러 누적·Slow3G. 판정: 로딩5초↓&전환2초↓=PASS / 10초↑ or 프리즈반복=보통 / 크래시=심각. 전제=E-018 카드 오진입 해결.

## 3. User 관점 — 설득력·완성도

### 3-A. 확보 결함 보고서 문구 (기대 vs 실제 vs 피해)
- **E-001**: 변경 없으면 경고 없이 닫혀야 → 무조건 경고. 피해=가짜경고 반복이 진짜 경고를 무력화(개선, 경미상향 여지).
- **E-014**: 헤더/탭에 씬 타이틀만 노출돼야 → 원본 파일명(`X3_F-1(2).jpg`) 노출. 피해=촬영장비·세트·순서 유추, 링크 유출 시 확대. 경미(공개뷰어 확인 시 보통, R2로 개발사 확인 후).
- **핫스팟 수정불가(D3-N2)**: 편집 가능해야 → 삭제 후 재등록만. 피해=정교배치 위치 재현 어려움(개선, 설계누락 성격).

### 3-B. 피드백 부재 테마 ("작업 성공/실패를 알 수 없는 지점들")
E-013(예약 미선택 저장 무반응·✅재현)·E-012(업로드직후 빈상태)·E-016(삭제직후 검은화면)을 **"작업완료 피드백 공통규약 부재"라는 시스템적 UX 공백**으로 묶어 요약/총평에 배치 → 경미 여러건을 설득력 있는 개선 테마로 승격. (E-006/E-007은 정상화로 제외)

### 3-C. 프라이버시 확장
회사목록 일간방문자 노출(회사간 통제?), 공유링크 접근통제 부재(공유=완전공개?), cntId 순차채번(존재 노출), 콘솔 내부정보(vtapi 호스트·jobId), 카테고리 폴백에 스페이스명 노출(E-005 연계).

### 3-D. 실사용 워크스루 (이탈 밀집 = 4.업로드~5.핫스팟)
업로드 구간: 규격 안내부재(E-004)·안내5MB vs 실제24MB+(E-010)·업로드직후 빈상태(E-012). 핫스팟 구간: 우클릭 발견성·도움말 불일치(E-003)·예약 미선택 무반응(E-013)·수정불가(D3-N2). 미검증 공백=반응형(창축소)·뷰어 줌 조작감 → D4 실측 권장(예측 체크리스트로만, 결함 단정 금지).

1. **업로드 세션(사람)**: E-008(최우선)→E-010 이진탐색→E-009 비율→포맷 페어와이즈→E-012 관찰. F12 상시.
