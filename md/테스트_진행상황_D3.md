## 3-3. 업로드 실측 (사용자 협업) — ★ 보통급 결함 확정 (통제 실험)

### D3-N5 (신규, 보통) — 특정 해상도 파노라마 업로드 처리 실패
- **현상**: 빈 스페이스 `업로드검증`(cntId=169)에 `X3\용량경계\` 이미지 업로드 → **06MB(7180×3590)·25MB·30MB(11968×5984)만 통과**, 08~20MB(전부 10771×5385)는 실패.
- **메커니즘(콘솔)**: 모든 업로드가 `[sceneUpload] jobId=...` 발급(=서버가 파일 수용, 비동기 파노라마 처리 시작) → **처리 단계에서 실패**. 전송 실패 아님.
- **근본원인 = 해상도 (통제 실험으로 증명)**:
  | 실험 | 콘텐츠 | 해상도 | 결과 |
  |---|---|---|---|
  | 원본 06MB | A콘텐츠 | 7180×3590 | 통과 |
  | 원본 08MB | B콘텐츠 | 10771×5385 | 실패 |
  | **검증A** | A콘텐츠(통과였던 것) | **10771×5385로 변경** | **실패** |
  | **검증B** | B콘텐츠(실패였던 것) | **11968×5984로 변경** | **통과** |
  → **콘텐츠 고정·해상도만 스왑 시 성패가 반전** = 해상도가 유일 원인. 용량(6MB통과/8MB실패/24MB통과)·비율(전부 2.00)·홀짝(짝수 10770×5384도 실패) 전부 반증.
- **실패 해상도 대역**: 9999×4999 / 10770×5384 / 10771×5385 (≈1만×5천) 실패. 7180×3590·11968×5984 통과. (정확한 임계 해상도는 개발사 확인/추가 이진탐색 영역)
- **피드백**: 실패 시 **"업로드 실패" 토스트는 표시됨**(2026-07-15 사용자 확인). 단 **왜 실패했는지·허용 해상도 안내는 없음**, 업로드 전 사전검증도 없음.
- **부수 결함**: ①씬 실패해도 **빈 유령 카테고리 생성**(E-011: `08MB_짝수`·`대조_홀수`) ②상시 콘솔 에러(E-015) 동반.
- **심각도**: **보통(Major)** — 핵심 기능(파노라마 업로드)이 정상 규격(2:1·유효 JPEG·적정 용량) 입력에서 특정 해상도라는 이유로 실패. (실패 토스트는 있으나 원인·허용범위 안내 없음)
- **증적**: 콘솔 jobId 다수 vs 씬 3개만 등록, 층수(카테고리) 다이얼로그의 유령 카테고리, 통제 실험 A/B 결과.
> E-008(배치 크래시 white screen)은 이 세션에서 **미발생** — 대신 위 해상도 결함으로 보통급 확보.

  | **WebP** | **실패** |
  | **GIF** | **실패** |
  → 해상도 동일·용량 동일 = **실패는 오직 포맷 때문** (허용: JPEG·PNG / 거부: WebP·GIF).
- **결함의 본질 = affordance 불일치 + 피드백 부재 (DOM 증거)**:
  - 파노라마 씬 업로드 input: **`accept="image/*"`** → 사용자가 WebP·GIF·BMP 등 **모든 이미지를 선택할 수 있게 열어둠**.
  - 같은 앱 다른 업로드 input: **`accept="image/png,image/jpeg"`** → 여긴 올바르게 제한. **입력마다 정책 불일치.**
  - 서버는 JPEG/PNG만 처리하고 WebP/GIF는 거부(콘솔 빈 jobId 2건). **"업로드 실패" 토스트는 표시됨**(사용자 확인) — 단 미지원 포맷이라는 사유는 안내 안 됨.
- **판정 논리**: "any image OK"처럼 보이는 UI(image/*)로 WebP/GIF 선택을 유도해놓고 처리는 못함 = 사용자 혼란(왜 실패인지 모름). 다른 input의 올바른 제한(`image/png,image/jpeg`)과 대비되는 **입력 간 정책 일관성 결여**. 경미~보통.
- **증적**: input accept 속성 스냅샷(`image/*` vs `image/png,image/jpeg`), 콘솔 빈 jobId 2건, 그라디언트 JPG/PNG 씬만 등록된 스크린샷.

### ⭐ D3-N7 (신규, 보통·강력) — 배치에 미지원 포맷 1장만 섞여도 정상 파일까지 전량 실패 (데이터 손실)
> ⚠ **1차 결론 정정**: 처음엔 "포맷거부 ∧ 해상도실패 공존 시에만 몰살"로 봤으나, 근거였던 "포맷테스트에서 jpg/png는 성공"이 **배치가 아니라 개별(하나씩) 업로드**였음이 사용자 확인으로 판명. → 배치 데이터에서 제외하고 재정리한 결과 아래 **단순·강력 법칙**으로 수렴.
- **통제 실험 매트릭스** (전부 배치=다중선택 1회 업로드, 사용자 실측 2026-07-15):
  | # | 배치 구성 | 거부(webp/gif) | 해상도실패 | 결과 |
  |---|---|:---:|:---:|---|
  | 1 | 정상 5장 / 정상 10장 | ✗ | ✗ | ✅ 전부 성공 (개수·동시성 무죄) |
  | 2 | 2장(통과1+해상도1) | ✗ | O(1) | ✅ 통과분 생존 |
  | 3 | 4장(통과2+해상도2) | ✗ | O(2) | ✅ 통과분 생존 |
  | 4 | 9장(통과3+해상도6) | ✗ | O(6) | ✅ 통과분 생존 (해상도실패 개수 무죄) |
  | 5 | **거부단독 3장(통과2+거부1)** | **O(1)** | ✗ | ❌ **전부 몰살** ← 거부 1장 단독으로도! |
  | 6 | 최소재현 4장(통과2+해상도1+거부1) | O(1) | O(1) | ❌ 전부 몰살 |
  | 7 | 10장(통과4+해상도4+거부2) | O(2) | O(4) | ❌ 전부 몰살 |
  | 8 | 12장 뒤섞기(통과5+해상도5+거부2) | O(2) | O(5) | ❌ 전부 몰살 |
- **결론(단일 법칙)**: **배치에 거부 파일(WebP/GIF 등 미지원 포맷 = `jobId 빈값`)이 1장이라도 있으면 정상 파일까지 전부 업로드 실패.** 해상도실패는 이 효과 없음(#2~4 통과분 생존). 파일 개수·용량·동시성·순서·해상도실패 개수 전부 무죄. 범인은 **거부 파일 단독**.
- **거부 대상 포맷**: WebP·GIF 확인(빈 jobId). 선택창 `accept="image/*"`로 허용하나 씬 업로드는 JPEG/PNG만 처리(D3-N6). BMP 등도 동일 추정.
- **추정 메커니즘**: 거부는 `jobId = (빈값)` 반환 → 빈 jobId를 폴링·집계하는 배치 완료 핸들러가 깨지며 **이미 성공한 씬 등록까지 롤백** → 정상 파일 유실.
- **심각도**: **심각(Critical)** [최종 확정, 사용자 승인 2026-07-15] — 데이터 손실 + 재현 100% + 통제실험 8종 방어. 실사용(폴더 통째 드래그에 스크린샷 WebP 1장만 섞여도) **멀쩡한 사진 전부 유실**. 순서 무관, **거부 1장으로 트리거**. (xlsx ID = **E-018**)
- **증적**: 매트릭스 8종, 콘솔 빈 jobId, "업로드 실패" 토스트, 빈 "뒤죽박죽/크래시테스트" 유령 카테고리(E-011), 개별 재업로드 교차확인.
> ※ 부수: `06_초소형_100x50.jpg`(100×50 초소형) 씬은 뷰어에서 **전체 검은화면** 렌더(별도 관찰, 스페이스 "크래시_재현_단일").
