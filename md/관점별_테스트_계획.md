## 0. 반드시 지킬 규칙 (모든 관점 공통 — test_rule.md + 양식규칙)

이 규칙을 어기면 서약 위반이거나 점수 감점이므로 **어떤 관점의 테스트든 아래를 먼저 통과해야 실행**한다.

| # | 규칙 | 실무 준수 방법 |
|---|---|---|
| R1 | **company 최대 10개** (현재 **2/10**) | 회사는 **신규 생성 지양, 기존 `회사01/02` 재사용**. 회사 생성 결함 검증은 1슬롯만. |
| R2 | **Space active = OFF 유지** | 발행/외부노출 테스트는 active를 켜지 않고, "active OFF 상태의 동작"만 검증. 발행 뷰어 확인이 꼭 필요하면 **개발사 확인 후에만**. |
| R3 | **테스트 용도 한정 · 파괴 금지** | 대량요청·실제 익스플로잇·타 사용자 데이터 변조 금지. 결함은 **재현+증적**으로 증명하되 서비스는 실제로 망가뜨리지 않는다. |
| R4 | **입력검증 공격은 순한 값만** | XSS는 `<b>텍스트</b>` 수준까지만. 파괴적 스크립트/SQL 시도 금지. |
| R5 | **사람이 해야 하는 작업** | ①로그인 비번 입력 ②5MB+·네이티브창 업로드 ③드라이브 다운로드 ④제출 메일 발송. 나머지 UI 조작·기록은 자동 진행. |
| R6 | **보고서 양식 불변** | `붙임2` xlsx 표 서식 변경 금지, **행 추가·크기조정만**. 줄바꿈 ALT+ENTER, 결함 1건=ID 1개. |

> 관점이 달라도 이 6개는 고정. 특히 **R1·R2**는 무계획 반복 시 슬롯 소진/서약위반으로 직결되니 각 관점 실행 전 체크.

## 1. 세 관점의 정의 — 무엇을, 왜, 어느 배점을 노리나

같은 화면을 보되 **질문이 다르다.** 세 관점이 겹치지 않게 역할을 나눠 커버리지 구멍을 없앤다.

| 관점 | 핵심 질문 | 주 무기 | 노리는 배점 | 대표 결함유형 |
|---|---|---|---|---|
| **QA** (품질보증) | "명세대로 동작하고, **빠짐없이** 덮었고, 산출물이 규칙을 지키는가?" | 명세 대조·정합성·커버리지 표·공식기법 설계 | 커버리지10 + TC완성도10 + 보고서완성도10 + 공식기법10 = **40** | 명세 불일치·상태 정합성·표기 오류 |
| **TE** (테스트 엔지니어) | "이 시스템을 **기술적으로 부술** 경계·보안·성능 취약점은?" | 경계값·API주입·퍼징·콘솔·네트워크주입·성능 | **결함발견 50** + 창의기법10 | Breakdown 크래시·검증 부재·IDOR·콘솔에러 |
| **User** (최종 사용자) | "실제 사용자가 겪는 **불편·혼란·불신**은?" | 실사용 시나리오·UX·피드백·접근성·프라이버시 | 결함(경미·개선) 기여 + 보고서 설득력 | 안내 부재·피드백 없음·프라이버시 노출·불필요 경고 |

> **분업 원칙**: TE가 "심각·보통"(50점 핵심)을 사냥, QA가 "빠짐없음+명세일치+산출물"을 굳히고, User가 "설득력 있는 경미·개선"으로 완성도와 폭을 채운다. 세 관점 결함이 **서로 다른 근본원인**이면 결함 총합이 saturate돼 50점에 가까워진다.

## 2. 관점별 상세 계획

### 2-A. QA 관점 — "빠짐없이 · 명세대로 · 규칙대로" (배점 40 방어)

**목표**: 결함을 못 찾아도 확보되는 안전점수 40을 만점으로 굳힌다.

**A-1. 커버리지 완성 (10점)** — 기능 인벤토리 전 소분류에 최소 1 TC, **PASS도 전부 기록**
- 기능트리(D1 확정본) 6대분류 각 소분류 매핑표 작성 → 커버리지 % 명시
- 아직 TC 없는 소분류 메우기: **비밀번호 찾기 / 내 정보(이름·전화 변경) / 플랜 조회 / 지도 좌표 / 공유·링크 / 뷰어 씬목록·지도토글**
- 산출: 「제품1」시트에 소분류별 PASS 케이스 추가(현재 PASS 19 → 목표 30+)

**A-2. TC 완성도 (10점)** — 모든 행이 "그대로 따라하면 재현되는" 품질
- 기대결과 구체화("정상 처리" 금지 → "메인 이동 + 상단 Login→로그아웃 전환" 식)
- 절차에 전제조건 포함, 각 TC에 **적용 기법 태그**
- D1 골격만 있던 행의 실제결과/판정 채우기

**A-3. 명세 정합성 결함 (QA가 잡는 결함)** — 이미 후보 확보, 재현·증적만
- E-002 토스트 띄어쓰기 불일치 / E-003 도움말 4종 vs 메뉴 5종 / E-005 카테고리 명명규칙 불일치
- 상태 정합성: **E-012 업로드 직후 빈 상태 표시(새로고침 필요)** / 씬 삭제→목록·뷰어 반영(D1 PASS 확인분) / E-016 대표삭제 렌더지연
- 추가 확인: **URL cntId 변경 시 콘텐츠 미전환**(라우팅 정합성) — 재현되면 결함

**A-4. 공식 기법 설계 산출물 (10점)** — [기술성_산출물.md](기술성_산출물.md) A절 이미 완료, 마감만
- 경계값표(용량·해상도) / 동등분할표(파일유형) / 결정표(접근권한) / 상태전이도(스페이스·씬 생명주기)
- **P1·P2 결과가 나오면 경계값표의 "확인중" 칸을 실측치로 확정** → 설득력 상승

**A-5. 보고서 완성도 (10점)** — R6 양식 준수 + 요약·환경·증적매핑
- 요약(환경·TC수·커버리지·심각도 분포) / 테스트 환경 명시(브라우저·OS·해상도·계정·기간)
- 「관련이미지」시트 결함ID 매핑(완료) + 오타·정렬 교차검수(7/18~19)

### 2-B. TE 관점 — "기술적으로 부순다" (결함 50 + 창의 10, 승부처)

**목표**: 심각·보통 결함을 **재현·증적 확실하게** 확보. 개발사 판정 방어가 생명.

**B-1. 업로드 Breakdown (최우선 · 심각/보통 밀집)** — 남은과제 P1~P3
- **P3 ★ E-008 크래시 격리(심각 후보)**: 빈 스페이스에 변형이미지를 **1장씩** 업로드해 white screen 유발 파일/조합 특정 → 재현절차 확정 + 콘솔(`null children`)·스크린샷 증적. **최우선.**
- **P1 E-010 용량 상한 이진탐색(보통, 확정)**: `X3\용량경계\` 06~24MB를 작은 것부터 업로드 → 통과↔거부 전환점 = 실제 상한. 이미 **24MB 통과 / 35MB 거부 / 안내 5MB** 확보 → 정확 상한값으로 강화.
- **P2 E-009 비율·해상도 검증부재(보통)**: 정사각/16:9/세로 각 단독 업로드 → 통과 시 왜곡/뷰어 검은화면, 검증 부재 확정. 100×50 검은화면(확보분) 증적.
- 포맷 개별: png(통과 확인)/webp/gif/bmp 단독 업로드 통과·차단 확정

**B-2. 인증·권한 (심각 유력 · 고배점)**
- **IDOR(수평 권한상승)**: UI에서 **관리 URL `cntId`를 타 스페이스 값으로 조작** → 남의 스페이스 접근되는지. (토큰 추출은 보류, UI 기반 재시도)
- API 직접 주입(창의 B-1, 완료): 토큰 없이 `content/auth?cntId=` 호출 → **401 확인(접근제어 정상, PASS 근거)**
- 로그아웃 후 뒤로가기로 관리화면 접근 / 세션 만료 후 저장·업로드 시도 처리

**B-3. 견고성·성능·콘솔**
- **E-015 콘솔 상시에러**: 빈/일반 에딧모드에서 `getVisibleChildId null children` 반복 → F12 Console 증적
- **P4 대형 투어(37씬) 성능**: H동 층별투어 진입(E-018 카드 오진입 먼저 해결) → 로딩시간·씬이동·프리즈·콘솔 측정
- 업로드 아웃티지 재확인: 재로그인 후 08MB 정상파일 다중 업로드 → 반복되면 "업로드 신뢰성" 결함, 1회성이면 환경으로 분리(섣불리 결함 단정 금지)

**B-4. 창의 기법 (10점)** — [기술성_산출물.md](기술성_산출물.md) B절, 2개 심화 완료
- API 주입(B-1) + 퍼징·SBTM(B-2) 방법·발견·판단 서술 / 페어와이즈(B-3) 조합표(45→9)

> **TE 수칙**: 발견 즉시 재현 재실행으로 확정 → 증적(스크린샷+콘솔) → 결함ID 부여. **심각도 과장 금지**(강등 시 신뢰 하락). R3·R4 준수(파괴 금지, `<b>`까지만).

### 2-C. User 관점 — "실사용자가 겪는 불편·불신" (완성도·폭 기여)

**목표**: 기술로는 안 보이지만 **사용자가 바로 느끼는** 결함으로 결함 폭과 보고서 설득력을 넓힌다. 대부분 경미·개선이나 **근거가 명확해 판정 방어가 쉽다.**

**C-1. 프라이버시·정보노출 (User가 가장 세게 주장 가능)**
- **E-014 원본 파일명 노출**: 에딧모드/뷰어 헤더에 `X3_H-1(1).jpg` 등 360카메라 원본 파일명 그대로 노출 → 촬영 위치·기기 유추 가능. **발행 공개 뷰어(로그아웃 상태)에서 최종 확정 시 경미~보통.** (R2 준수 위해 발행은 개발사 확인 후)

**C-2. 피드백 부재 (사용자가 "됐는지 안 됐는지 모름")**
- **E-006 공유 버튼 무반응**: 클릭해도 토스트·모달 없음 → 복사됐는지 사용자 알 수 없음
- **E-013 예약 핫스팟 구역 미선택 저장**: 아무 반응·안내 없음 → 저장 실패를 사용자가 인지 못함
- 공통 패턴: **"작업 결과 피드백 없음"** → 사용성 개선사항으로 묶어 서술 가능

**C-3. 불필요·혼란 유발 UX**
- **E-001 변경 없이 취소해도 "작성 중 내용 삭제" 경고** → 불필요한 불안 유발
- **E-007 등록 모달 재오픈 시 직전 입력값 잔존** → 새 등록인데 이전 값 → 실수 유발
- **E-004 업로드 용량/포맷 안내 부재**: 사용자가 규격을 알 수 없어 시행착오 (E-010과 연결 — 안내 5MB인데 실제 24MB+ 통과 = 사용자 혼란)

**C-4. 실사용 시나리오 워크스루 (신규)**
- "처음 온 사용자가 회사→스페이스→이미지 업로드→씬 구성→핫스팟→(발행 전) 확인"까지 **막힘 없이 되는가**를 유스케이스로 1회 관통 → 중간 이탈지점·안내부족 기록
- 반응형/창 축소 시 씬목록 패널 레이아웃, 뷰어 회전·줌 조작감

> **User 수칙**: 결함이라 우기지 말고 **"사용자 기대 vs 실제"**로 서술. 경미·개선이라도 커버리지·완성도·창의성(사용자 시나리오 기반 탐색)에 기여한다.

## 4. 관점 → 배점 → 산출물 연결 (한눈에)

| 관점 | 배점 기여 | 산출물 반영 위치 |
|---|---|---|
| QA | 커버리지10+TC10+보고서10+공식기법10 = **40** | 「제품1」PASS 케이스·기법태그, 「작성법」규칙준수, 기술성 A절 |
| TE | **결함 50** + 창의10 | 「제품1」심각·보통 FAIL 행, 「관련이미지」증적, 기술성 B절 |
| User | 결함(경미·개선) + 완성도 설득력 | 「제품1」경미·개선 FAIL 행, User 시나리오 세션노트 |
| 가산점 | +4~6 | 재학증명(2) + 수료증(장당1) — 시험장 밖 확정점 |

> **결론**: QA로 40을 굳히고(안전빵), TE로 50에 화력 집중, User로 폭·설득력·완성도를 더한다. 세 관점 모두 **R1~R6 규칙 안에서** 실행 — 이게 판정 방어와 서약 준수를 동시에 만족시키는 유일한 길.
