## 0. 제품 구조 (직접 확인한 실제 구조)

VT partners는 **360 VR 가상투어(VT)를 UI로 직접 만들고 관리하는 웹 저작·배포 도구**입니다. 크게 세 영역으로 나뉩니다.

| 영역 | URL | 역할 |
|---|---|---|
| **관리자(Admin)** | `roundpic.io/#/login` → `VT PARTNERS Admin` | 이메일/비밀번호 로그인, 파노라마 업로드·VT 생성·핫스팟 편집·프로젝트 관리 (**테스트 계정 필요**) |
| **갤러리(Main)** | `roundpic.io/#/main` | PROJECT LIST — 제작된 VT 사례 목록 |
| **뷰어(Viewer)** | `roundpic.io/{프로젝트명}` 또는 `/#/vt/{id}` | 완성된 VT를 일반 사용자가 보는 화면 (로그인 불필요) |

### 뷰어에서 직접 확인한 기능 (계정 없이도 테스트 가능)
- **360 파노라마 드래그 회전 / 줌** — equirectangular(2:1) 파노라마 이미지 기반
- **씬 이동 핫스팟** — 원형 마커 + 라벨(예: "침실")을 클릭하면 다른 씬(장면)으로 전환
- **씬 목록 패널** — 우측 화살표(◀)로 펼침, **카테고리 탭**으로 그룹화됨 (예: "최종테스트 / 삼각대")
- **썸네일 네비게이션** — 각 씬 썸네일 클릭으로 이동
- **지도(Map) 버튼** — 우상단, 지도 오버레이 토글
- 씬 상단에 **파일명 표시** (예: `IMG_20260701_102928_00_764.jpg` — 360 카메라 원본 파일명 그대로 노출됨 → **개인정보/UX 결함 후보**)

### 관리자에서 예상되는 기능 (계정 발급 후 테스트할 핵심)
- 로그인 / 비밀번호 찾기 / 솔루션 이용 문의
- 프로젝트(VT) 생성·수정·삭제·복제·목록
- **파노라마 업로드** (여기가 결함 최다 예상 지점)
- **씬 구성** + **씬 간 이동 연결(핫스팟 배치)**
- **인터랙션 핫스팟 3종**: 사진 / 동영상 / **예약**
- 카테고리(그룹) 관리, 지도 좌표 설정, 발행/공유

## 1. 360 파노라마 이미지 준비 — ✅ 확보 완료, 변형본만 제작

공고문에도 **"제품 특성상 360 파노라마 이미지 필요"**라고 명시돼 있고, 실제 뷰어도 equirectangular 2:1 파노라마를 올려야 씬이 만들어집니다.

### ① ✅ 정상 규격 이미지 — 확보 완료 (드라이브)
- [360 테스트 파노라마 드라이브](https://drive.google.com/drive/folders/1fcxn6p1wyWx3-TDWA_DBUY4DWi5uWK8i) 제공됨.
- **첫 작업**: 드라이브 내용 확인 → 정상 규격 1~2장으로 **VT 1개를 끝까지 생성**(업로드→씬→핫스팟→발행) = **PASS 기준선 + 조작법 습득**.
- 개발사 제공본이라 실제 서비스 기대 규격 → 정상 케이스 기준으로 사용.

### ② 경계/이상 변형본 제작 (결함 사냥 탄약 — 여기에 시간 투자)
정상 파노라마 1장을 기준으로 **일부러 규격을 벗어난 이미지**를 만들어 업로드 결함을 유발. (무료 소스: Pexels/Wikimedia Commons에서 추가 2:1 파노라마 확보 가능)
- Pexels, Wikimedia Commons 등에서 2:1 파노라마 무료 다운로드 (아래 출처 참조)
- 규격 표준: equirectangular는 **가로:세로 = 2:1** (예 4096×2048, 8192×4096)

### ③ 직접 촬영 (3순위 — 여력 있으면)
- 360 카메라(Ricoh Theta, Insta360) 또는 스마트폰 "Google 스트리트뷰" 앱의 포토스피어 기능으로 직접 촬영 → 실사용 시나리오 재현. (실제 서비스 파일명이 `..._00_764.jpg` 형태라 Ricoh Theta 계열 원본으로 추정됨)

### ④ 경계값·이상 이미지 세트 미리 제작 (결함 사냥 탄약)
정상 파노라마 1장을 기준으로 아래 변형본을 **미리 만들어 두면** 업로드 결함을 체계적으로 팔 수 있음:

| 변형 | 목적 |
|---|---|
| 초대용량 (예: 50MB↑, 8K↑) | 용량 상한/타임아웃 |
| 0바이트 / 손상 파일 | 오류 처리 |
| 2:1 아닌 비율 (정사각, 16:9, 세로) | 파노라마 왜곡/검증 여부 |
| 지원 외 포맷 (.png, .webp, .heic, .gif, .bmp) | 포맷 필터링 |
| 확장자 위장 (.txt를 .jpg로) | 실제 검증 여부 (보안) |
| 초소형 (100×50) | 하한 처리 |
| 한글·특수문자·초장문 파일명, 이모지 | 파일명 처리 |
| EXIF 회전정보 있는 이미지 | 방향 처리 |

## 3. 테스트 기법 매핑 (기술성 20점: 공식 10 + 창의 10)

각 TC 옆에 "적용 기법" 열을 만들어 아래를 명시. **기법 이름만 적지 말고 "왜 이 기법인지" 한 줄 근거**를 달 것.

### 공식 테스트 기법 (10점)
| 기법 | 적용 예시 |
|---|---|
| 동등분할 | 로그인 아이디: 유효/무효/미가입 그룹 |
| 경계값 분석 | 업로드 용량 상·하한, VT 제목 글자수, 예약 날짜 경계 |
| 결정 테이블 | (로그인 여부 × VT 공개상태 × 소유자) → 접근 가능/불가 |
| 상태 전이 | VT: 작성중 → 발행 → 수정 → 삭제, 세션: 로그인 → 만료 |
| 명세 기반 조합 | 핫스팟 종류 × 대상 씬 존재여부 |

### 창의적 기법 (10점) — 1~2개를 깊이 있게
| 기법 | 설명 |
|---|---|
| **페어와이즈(Pairwise) 조합** | 브라우저(3) × 해상도(3) × 핫스팟종류(3) 전조합 대신 페어와이즈로 축소 설계 → "적은 케이스로 넓은 커버리지" 논리로 서술 |
| **네트워크 조건 주입(개발자도구)** | Chrome DevTools로 오프라인/Slow 3G/스로틀링 상태에서 대용량 파노라마 로딩·업로드 → 실사용 회선 재현 |
| **SBTM(세션 기반 탐색적 테스트)** | 90분 세션 차터("업로드 기능을 30분간 파괴해본다") 운영, 세션노트·발견결함을 기록으로 첨부 → 정성 평가에 강함 |

## 4. 결함 사냥 시나리오 (50점 — 심각도×배점, 여기가 승부처)

실제 웹 VR 저작도구의 구조상 결함이 잘 나오는 지점을 우선순위로 정리. **재현절차가 명확하고 증적(스크린샷/영상)이 확실한 것만** 보고 (결함 여부는 개발사 상의 후 판정되므로).

### A. 파노라마 업로드 (최우선 — 결함 밀집)
- 초대용량/0바이트/손상 파일 업로드 시 오류 처리·안내 메시지
- 2:1 아닌 비율 업로드 시 왜곡되는가 / 검증하는가
- 지원 외 포맷, 확장자 위장 파일 통과 여부
- 업로드 진행 중 페이지 이탈·새로고침 시 상태
- 다중 업로드 시 순서·개수 제한·부분 실패 처리

### B. 인증·권한 (심각도 높음 → 고배점)
- 로그아웃 후 **뒤로가기**로 관리자 화면 접근되는지
- 뷰어/관리 URL의 **VT id를 바꿔** 남의 비공개 VT 접근 (수평 권한상승, IDOR)
- 세션 만료 후 저장/업로드 시도 시 처리
- 로그인 입력값: 빈값, 공백만, 대소문자, 초장문, 앞뒤 공백 트림 여부

### C. 씬·핫스팟 상태 꼬임
- 다른 씬으로 **이동 핫스팟**을 걸어둔 뒤 **대상 씬을 삭제** → 깨진 링크 처리
- 편집 중인 씬을 다른 탭에서 삭제 (동시 편집 충돌)
- 핫스팟을 화면 밖/겹치게 배치, 좌표 경계
- 예약 핫스팟: **과거 날짜 예약**, 자정·월말 경계, 빈 값, 동시 예약

### D. 입력값 검증
- VT 제목/씬 이름/설명: 최대 길이 초과, 이모지, 앞뒤 공백, HTML 태그(`<b>`텍스트 정도로만 — **파괴적 스크립트 시도 금지**, 계정은 테스트 용도만)
- 지도 좌표: 범위 밖 위경도, 문자 입력

### E. 뷰어/UI/호환
- 씬 목록 패널 열고 창 크기 축소 시 레이아웃 깨짐
- Chrome / Edge / Firefox 간 렌더링·핫스팟 동작 차이
- 모바일 해상도(반응형) 동작
- **콘솔 에러**(DevTools Console) — 눈에 안 보여도 에러 로그는 결함 근거
- 파일명이 씬 상단에 그대로 노출되는 UX/프라이버시 이슈

### 운영 방식
- **2인 역할 분담**: A=조작·탐색 / B=즉시 기록·스크린샷 (라운드 로빈으로 교대)
- 결함 발견 → **즉시 재현 절차 재실행하여 확정** → 증적 캡처 → 결함ID 부여(E-001…)
- **심각 결함 소수 > 사소 결함 다수** (심각도 배점이 곱해짐). 단 사소(개선사항)도 커버리지·완성도에 기여하니 전부 기록.

## 5. 심각도 판정 기준 (보고서 양식의 정의 그대로)

양식 "작성법" 시트 기준:

| 심각도 | 정의 | 예시 |
|---|---|---|
| **심각(Critical)** | 해당 기능 및 이후 기능들이 진행 불가한 Breakdown | 업로드가 아예 안 됨, 로그인 불가, VT 저장 안 됨 |
| **보통(Major)** | 기능은 동작하나 명세와 다르게 동작 | 삭제한 VT가 목록에 남음, 핫스팟이 엉뚱한 씬으로 이동 |
| **경미(Minor)/개선** | 사용에 큰 지장 없는 사소한 결함·개선사항 | 오타, 정렬 어긋남, 안내문구 미흡 |

> 심각도가 높을수록 배점이 곱해지므로, **A(업로드)·B(권한) 영역의 심각 결함**을 최우선으로 노려야 50점에 근접함.

## 6. 결과보고서 작성 규칙 (양식 준수 = 완성도 10점) — 실제 양식 확인본

`붙임2. SW제품 테스트 결과보고서(양식).xlsx` — 시트 3개, **표 양식 변경 불가 / 행 추가·크기조정만 허용.**

**① 「보고서 작성법」 시트** (규칙 — 반드시 정독)

**② 「제품1」 시트** — 실제 열 구조 (A~J):

| A | B | C | D | E | F | G | H | I | J |
|---|---|---|---|---|---|---|---|---|---|
| No. | 대분류 | 중분류 | 소분류 | 테스트 절차 | 기대 결과 | 실제 결과 | 결과(PASS/FAIL) | 결함 심각도 | 결함 ID. |

**③ 「제품1_관련이미지」 시트** — A `이미지 No` / B `결함 ID` / C `이미지` (본문 결함ID와 1:1 또는 1:N 매핑)

### 심각도 4단계 (양식 정의 그대로)
| 등급 | 정의 |
|---|---|
| **심각** | Breakdown(장애로 전체 기능 정지) 또는 비정상 종료 등 **즉시 수정 필요** |
| **보통** | 운영엔 문제없으나 **기능이 정확히 동작 안 하거나 수행결과≠예상결과** |
| **경미** | 지장 없으나 기대≠실제 (**표시 오류·레이아웃 불일치** 등) |
| **개선사항** | 결함은 아니나 사용 편의·개선 필요 (가독성·버튼 위치·불필요 UI 등) |

### 작성 원칙
- **PASS 케이스도 전부 기록** (FAIL만 적으면 커버리지 손해). PASS는 I열(심각도) `-`.
- 결함 1건당 결함ID 1개, **제품별 `XX-001`부터** (E-001, EC-001 등 자유). 한 기능에서 복수 결함 가능.
- 테스트 절차 = **이벤트 중심 동작순서 상세 기록**("그대로 따라 하면 재현되는" 수준, 전제조건 포함).
- 기대결과 = 정상 시 기대값(구체적으로, "정상 처리" 금지). 실제결과 = 실제 실행 관찰값(스크린샷 대응).
- **줄바꿈 = ALT + ENTER.** 증적 이미지는 결함ID와 매핑(용량 크면 별도 첨부, 이미지명·ID 일치).
