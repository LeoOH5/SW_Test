## 🔴 심각 (Critical)

### E-008 [수동] 변형 이미지 배치 업로드 중 에디터 white screen 크래시
- 빈 스페이스 에딧모드에서 변형 이미지 여러 장 배치 업로드 → **하얀 빈 화면**, 콘솔 `Cannot read properties of null (reading 'children')`. 새로고침 복구. 단일 파일 미재현 → 간헐.
- 증적: white screen + F12 콘솔. **재현조건(몇 장부터/조합) 좁히기 진행 중.**

### A-01 [자동] 비활성(미게시) 스페이스 콘텐츠가 비인증 뷰어 API로 유출 ★신규·직접 실측
- `active=false`(미게시) 스페이스에 씬 생성 후 **토큰 없이** `GET vtapi.visitinc.kr/vt?VT=153dfe4` → **HTTP 200으로 전체 콘텐츠 서빙**(스페이스 제목·씬 정보·**360 타일 저장경로**).
- **의미**: 게시 여부와 무관하게 **VT 키만 알면 누구나 미게시 스페이스 열람**. 키는 7자리 hex라 추측·열거 리스크도 있음.
- **주의**: 수동 테스트는 관리자 엔드포인트(`content/auth?cntId=`)를 토큰 없이 호출해 401을 확인하고 "접근제어 정상"으로 판정했으나(정당), **뷰어 데이터 엔드포인트(`/vt?VT=`)는 별개로 미게시 콘텐츠를 노출**한다. 접근제어 결론을 부분 보정하는 결함.
- 증적: verify_v1 실측(사전 active=false 확인 → 씬 생성 → 비인증 200·콘텐츠·타일경로 노출 → 정리). 게시상태 게이팅 필요.

## 🟠 보통 (Major)

### E-009 [양쪽] 초소형(100×50) 이미지가 씬으로 통과 → 뷰어 검은 화면
- 최소 크기/비율 검증 부재. **[자동]** API에서 tiny가 실제 씬으로 생성됨 확인 + **[수동]** 뷰어에서 완전 검은 화면 육안 확인. 안내·플레이스홀더 없음.

### E-010 [수동] 업로드 용량 상한이 안내와 5배+ 불일치
- 안내는 "5MB까지"인데 **실측 24MB까지 통과**(25/30MB.jpg=실24MB), 35MB 거부. 상한 미정의·안내 오류. (자동 테스트는 압축으로 파일이 작아 이 결함을 못 잡음 — 수동의 고유 성과)

### E-012 [양쪽] 업로드 직후 에디터가 "이미지 없음"으로 잘못 표시, 새로고침 필요
- 비동기 job 반영 지연. **[수동]** 육안 확인 + **[자동]** 이 현상이 자동 렌더 캡처를 막았음(같은 결함을 양쪽이 독립 확인).

### A-02 [자동] 로그인 실패(미가입 이메일)에 HTTP 201 + "id not exist" → 성공코드화 + 사용자 열거
- 정상형식 미가입 이메일+임의 비번 → **201 + `{"message":"id not exist"}`**. ① 인증 실패를 2xx로 통보(401이어야) ② 가입/미가입 계정 구분 노출 = 사용자 열거. 수정: 401 + 일반 메시지.

### A-03 [자동] 입력검증 미비로 잘못된 요청이 4xx 대신 HTTP 500
- 초장문 파일명 업로드(UTF-8 ~255바이트 초과, 한글 100자↑) → `POST /scene/autoNorth` **500**(임계 50자 202/100자 500). PUT /content 불완전 body → **500**(생성 POST /content는 400 검증 존재 → 검증 비대칭).

### A-04 [자동] 보안 응답 헤더 6종 전무 + 인증 토큰 localStorage 평문 저장
- CSP·X-Frame-Options·HSTS·X-Content-Type-Options·Referrer-Policy·Permissions-Policy 전부 부재(web+API 호스트). `x-powered-by: Express` 노출. 토큰이 httpOnly 쿠키 아닌 **localStorage 평문** → CSP 부재와 결합 시 XSS로 세션 탈취.

## 🔗 교차검증 하이라이트 (두 접근이 서로 확인)

1. **E-012 ↔ 자동 렌더 실패**: 수동이 "업로드 후 새로고침해야 반영"을 결함으로 확인. 자동 테스트에선 이 현상이 360 렌더 캡처를 막았음 → **제품 결함으로 확정**(도구 문제 아님).
2. **업로드 아웃티지(요확인) ↔ 짧은 토큰 TTL**: 수동이 "세션 후반 정상 파일도 업로드 실패, 재로그인 복구"를 관찰. 자동도 **토큰이 몇 분 만에 만료**되는 것을 확인. 독립 발견 → "업로드/세션 신뢰성" 결함 후보.
3. **E-009 tiny / E-014 파일명 / 콘솔에러**: 수동 육안 + 자동 관찰이 동일 대상을 각각 확인.

## 방법·환경·정직성
- 수동: Chrome/Windows, 실사용 시나리오·육안·콘솔. 자동: Playwright(헤드리스+Aside CDP 실브라우저), 실 API 관찰, fable 매니저-워커 E2E(후보 61→검수 승인) + Opus 최종 검수.
- 증거 원본: (자동) recon/*.json·verify_v1, (수동) 진행상황/기술성/증적 md + `SW제품 테스트 결과보고서_작성본.xlsx`.
- **정직성**: 재현·증적 확실한 것만 보고. 추정은 "추정" 명기. 검증 아티팩트(예: DELETE HTTP-only 오판, E-019 다중누락)는 철회/기각. 간헐 결함(E-008)은 간헐로 표기. 최종 심각도는 개발사 상의 후 확정.
- **제출 반영 권장**: A-01(심각)·A-02·A-03·A-04를 xlsx 「제품1」 시트에 결함ID로 추가(양식 서식 불변·행 추가만·PASS 포함·ALT+ENTER). A-01은 심각도·임팩트가 커 개발사 우선 보고 대상.
